## Supabase Row Level Security

**Severity:** MEDIUM  
**Check ID:** `supabase_rls`

**Summary:** 3 tables not RLS-enabled in code: comments, posts, users

### Why this matters

These tables are created in your SQL migrations but we found no `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` and no `CREATE POLICY ... ON <table>` for them in any `.sql` file.

**Two possible meanings:**

1. **RLS actually IS off.** Anyone with your anon key (which is public in your frontend bundle) can read every row of these tables. This is a real and common Supabase mistake.

2. **You enabled RLS via the Supabase dashboard** (toggling RLS in the Table Editor or Authentication → Policies). The setting lives in your database, not your code — so a static check like this can't see it.

**How to tell which it is:** open Supabase → Table Editor → click each table → look at the top of the table's page. If you see a shield icon and "RLS enabled", you're in case (2) and this finding is a false positive — see the fix section for how to silence future audits.

### Where

- `supabase/migrations/001_init.sql:4` — Table `users` is defined but RLS is never enabled.
  ```
  CREATE TABLE users
  ```
- `supabase/migrations/001_init.sql:11` — Table `posts` is defined but RLS is never enabled.
  ```
  CREATE TABLE posts
  ```
- `supabase/migrations/001_init.sql:20` — Table `comments` is defined but RLS is never enabled.
  ```
  CREATE TABLE comments
  ```

### How to fix

**If RLS is actually OFF on these tables** (verify in Supabase first):

```sql
ALTER TABLE your_table_name ENABLE ROW LEVEL SECURITY;

-- Deny everything by default for the anon role
CREATE POLICY deny_anon ON your_table_name
  AS RESTRICTIVE FOR ALL TO anon USING (false);

-- Then explicitly allow what authenticated users can do, e.g.:
CREATE POLICY users_can_read_own ON your_table_name
  FOR SELECT TO authenticated USING (auth.uid() = user_id);
```

**Test it:** in your browser console, try reading the table with just the anon key. You should get zero rows back.

**If RLS is already ON (enabled via the dashboard):** good — there's no real vulnerability. But future audits will keep flagging these tables. To silence the warning AND track your RLS config alongside your code, add the matching `ALTER TABLE` and `CREATE POLICY` statements to your migration files. This is good practice anyway: if you ever need to recreate your database, the dashboard settings won't come with you, but migration files will.

**Verify in Supabase**: Table Editor → click table → look for the RLS shield icon at the top. Or run `SELECT relname, relrowsecurity FROM pg_class WHERE relname = 'your_table_name';` — `relrowsecurity = true` means RLS is on.

## SQL injection risk

**Severity:** HIGH  
**Check ID:** `sql_injection`

**Summary:** 3 potential SQL injection point(s).

### Why this matters

SQL queries built by concatenating user input directly into the query string are vulnerable to SQL injection. An attacker can craft a request that breaks out of the intended query structure and runs arbitrary SQL.

Real-world impact:
- Dump every row of every table (`UNION SELECT * FROM users; --`)
- Modify or delete data
- Sometimes: gain shell on the database server

This is the most-scanned vulnerability class on the internet. Automated bots probe public sites with classic payloads (`' OR 1=1 --`) within hours of going live. If you're vulnerable, you WILL be hit.

### Where

- `queries.js:10` — SQL query built by interpolating request data — classic SQL injection vector.
  ```
  "SELECT * FROM users WHERE id = " + req.
  ```
- `queries.js:16` — SQL query built by interpolating request data — classic SQL injection vector.
  ```
  `SELECT * FROM posts WHERE title LIKE '%${req.query.q}%'`
  ```
- `queries.js:22` — SQL query built by interpolating request data — classic SQL injection vector.
  ```
  "DELETE FROM comments WHERE id = " + req.
  ```

### How to fix

**The universal fix: parameterized queries.** Never concatenate input — use placeholders and pass data separately. The database driver handles escaping.

**Node.js (pg):**
```js
// BAD
const result = await db.query(`SELECT * FROM users WHERE id = ${req.params.id}`);

// GOOD
const result = await db.query(
  "SELECT * FROM users WHERE id = $1",
  [req.params.id]
);
```

**Python (psycopg2 / asyncpg):**
```python
# BAD
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")

# GOOD
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))
```

**Even better: use an ORM** so this whole bug class becomes impossible:
- JS/TS: Prisma, Drizzle, Supabase JS client
- Python: SQLAlchemy, Django ORM, Supabase Python client

**For dynamic queries** (column names, ORDER BY direction) where placeholders don't work: validate against an allow-list. Never accept arbitrary user input.

## Missing authentication on API routes

**Severity:** HIGH  
**Check ID:** `missing_auth_on_routes`

**Summary:** 4 state-changing route(s) with no auth check.

### Why this matters

These endpoints accept POST/PUT/DELETE/PATCH requests and we found no authentication code in their file. Without auth, anyone on the internet can call them — which is fine for a public form but catastrophic for anything that modifies user data.

Worst cases:
- `DELETE /api/users/:id` with no auth → anyone deletes any account
- `POST /api/admin/...` with no auth → anyone runs admin actions
- `PATCH /api/users/me` with no auth → user identity comes from request body

### Where

- `routes/admin.js:6` — DELETE handler with no auth check anywhere in the file.
  ```
  DELETE /admin/users/:id
  ```
- `routes/admin.js:13` — POST handler with no auth check anywhere in the file.
  ```
  POST /admin/posts/:id/feature
  ```
- `routes/admin.js:20` — PUT handler with no auth check anywhere in the file.
  ```
  PUT /admin/settings
  ```
- `queries.js:21` — POST handler with no auth check anywhere in the file.
  ```
  POST /comments/delete
  ```

### How to fix

Add auth verification BEFORE the handler runs. Examples:

**Express + Clerk middleware:**
```js
import { clerkMiddleware, requireAuth } from "@clerk/express";
app.use(clerkMiddleware());

app.post("/api/protected", requireAuth(), (req, res) => {
  // req.auth.userId is the verified Clerk user ID
});
```

**FastAPI + dependency injection:**
```python
from fastapi import Depends, HTTPException

def get_current_user(authorization: str = Header(None)):
    if not authorization:
        raise HTTPException(401, "Missing token")
    # verify with Clerk JWKS or your auth provider
    return verified_user_id

@app.post("/protected")
async def protected(user_id: str = Depends(get_current_user)):
    ...
```

**Lambda handler:**
```python
def lambda_handler(event, context):
    token = event["headers"].get("authorization", "").replace("Bearer ", "")
    user_id = verify_clerk_token(token)
    if not user_id:
        return {"statusCode": 401, "body": "Not authenticated"}
    # safe to proceed
```

Public endpoints (`/health`, `/login`, webhooks) are intentionally exempt — this check already ignores those.

## Secrets in client-side code

**Severity:** HIGH  
**Check ID:** `secrets_in_client`

**Summary:** 1 secret(s) embedded in client-side code.

### Why this matters

A secret in client-side code is **public**. Every user who loads your site downloads the JavaScript bundle. Anyone with DevTools can read it. Once a secret is in the bundle, treat it as compromised — even if you delete it later, every user who already loaded the page has a copy.

The most dangerous cases:
- Stripe live/restricted keys → attacker can charge customers or refund themselves
- OpenAI/Anthropic keys → attacker racks up your bills
- AWS access keys → attacker can spin up resources, dump your S3 buckets
- Supabase service role → bypasses every RLS policy you wrote

### Where

- `public/checkout.html:10` — Found OpenAI API key in client-side code. Visible to every user in their browser.
  ```
  sk-ant-F...REDACTED
  ```

### How to fix

**Immediate steps:**
1. **Rotate every key shown.** Treat them as leaked.
2. **Move the key to server-side code.** Frontend should never need a secret; it should call your backend, and the backend uses the secret.

**Pattern: replace direct API call with proxy through your backend**

*Before (insecure, in React component):*
```jsx
const res = await fetch("https://api.openai.com/v1/...", {
  headers: { Authorization: "Bearer sk-..." }  // BAD
});
```

*After:*
```jsx
// React component just calls your own API
const res = await fetch("/api/chat", { method: "POST", ... });
```
```js
// /api/chat.js (server-side — env var lives here)
export default async function handler(req, res) {
  const openaiRes = await fetch("https://api.openai.com/v1/...", {
    headers: { Authorization: `Bearer ${process.env.OPENAI_API_KEY}` }
  });
  res.json(await openaiRes.json());
}
```

**For values that ARE meant to be public** (Stripe publishable key, Clerk publishable key, Supabase anon key with RLS): rename the env var with a `NEXT_PUBLIC_`, `VITE_`, or `REACT_APP_` prefix. That makes the "this is meant to be public" intent explicit, and Trendslop won't flag it.

## Rate limiting on auth endpoints

**Severity:** MEDIUM  
**Check ID:** `rate_limiting_auth`

**Summary:** 1 auth endpoint(s) without rate limiting.

### Why this matters

Without rate limiting, attackers can brute-force credentials, enumerate which emails have accounts (signup endpoints), or spam password reset emails. None of these requires sophisticated tooling — there are publicly available scripts that try millions of credential pairs per hour against an unprotected login endpoint.

### Where

- `server.js:12` — Auth-related endpoint with no rate-limiting library imported or applied.

### How to fix

Add per-endpoint rate limiting to login, signup, and password reset routes:

**Express (Node.js):**
```js
import rateLimit from "express-rate-limit";
const authLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 5,  // 5 attempts per IP per minute
  message: "Too many requests, try again later."
});
app.post("/login", authLimiter, loginHandler);
app.post("/signup", authLimiter, signupHandler);
app.post("/reset-password", authLimiter, resetHandler);
```

**FastAPI (Python):**
```python
from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)

@limiter.limit("5/minute")
@app.post("/login")
async def login(request: Request, ...): ...
```

**AWS Lambda + API Gateway:** add a Usage Plan with throttling, or use AWS WAF rate-based rules. In `serverless.yml`:
```yaml
provider:
  apiGateway:
    usagePlan:
      throttle:
        burstLimit: 10
        rateLimit: 5
```

## HTTPS-only enforcement

**Severity:** MEDIUM  
**Check ID:** `https_only`

**Summary:** 3 HTTPS issue(s) — cookies and/or hardcoded http:// URLs.

### Why this matters

Plain-HTTP traffic is readable by anyone on the same Wi-Fi network as your users. Even if you redirect to HTTPS, the first request still hits HTTP — meaning the connection downgrade attack ("SSL stripping") is possible.

Two common code-level mistakes:
1. **Cookies without `secure: true`** — the browser will send the cookie over plain HTTP if a downgrade happens, leaking session tokens.
2. **Hardcoded `http://` URLs in API calls** — every call leaks request/response data in plain text.

### Where

- `server.js:14` — Cookie set without the `secure` flag — will travel over plain HTTP if a downgrade happens.
  ```
  res.cookie('session_id', sessionId, { httpOnly: true, sameSite: 'lax', secure: process.env.NODE_ENV
  ```
- `server.js:15` — Cookie set without the `secure` flag — will travel over plain HTTP if a downgrade happens.
  ```
  res.cookie('auth_token', 'xyz', { maxAge: 86400000, httpOnly: true, sameSite: 'lax', secure: process
  ```
- `server.js:20` — Cookie set without the `secure` flag — will travel over plain HTTP if a downgrade happens.
  ```
  res.cookie('session_id', '', { maxAge: 0, httpOnly: true, sameSite: 'lax', secure: process.env.NODE_
  ```

### How to fix

**Cookies — always set `secure` and `httpOnly`:**

**Express:**
```js
res.cookie("session", token, {
  httpOnly: true,
  secure: true,           // HTTPS only
  sameSite: "strict",     // mitigate CSRF
  maxAge: 7 * 24 * 60 * 60 * 1000
});
```

**FastAPI:**
```python
response.set_cookie(
  key="session", value=token,
  httponly=True, secure=True, samesite="strict"
)
```

**Replace hardcoded `http://` URLs with `https://`** — except for localhost in development. If the URL is a config value, move it to an env var so dev and prod can differ:
```js
// .env.local: API_BASE=http://localhost:3001
// .env.production: API_BASE=https://api.yourapp.com
const apiBase = process.env.API_BASE;
```

**Add HSTS at the deployment layer** (CloudFront / Vercel / Nginx):
```
Strict-Transport-Security: max-age=31536000; includeSubDomains
```
